The feature runs through four layers, Task owns the data and lifecycle logic, TaskStorage syncs an in-memory dict to tasks.json, TaskManager validates inputs and orchestrates the two, and cli.py routes commands. Creation converts raw inputs to enums and datetimes, builds a Task with an auto-generated UUID, then rewrites the full JSON file. Status updates follow the same path except DONE, which bypasses the normal update to call mark_as_done() directly and capture completed_at. The custom encoder/decoder pair keeps serialization transparent, and that DONE inconsistency is worth watching if the update logic ever gets extended.
The flow enters through cli.py as status <id> done, passes to update_task_status() in TaskManager, which converts the string to a TaskStatus enum and takes a special path, it fetches the live Task object via storage.get_task(), calls mark_as_done() directly to stamp status, completed_at, and updated_at in memory, then storage.save() rewrites the entire tasks.json. The two quiet failure points are get_task() returning None for an unknown ID and save() failing silently, which can leave memory and file out of sync with no warning to the user.
Rewriting get_tasks_by_status to use a list comprehension was the clearest example of idiomatic Python, the one-liner reads like English and removes the need for a temporary variable entirely. The broader lesson is that Python rewards expressing what you want rather than how to build it, and returning directly from an expression is almost always cleaner than accumulating into a variable just to return it at the end.
Reviewing update_task_status revealed that the DONE path returns None implicitly when the task isn't found, while every other path returns an explicit boolean, a silent inconsistency that would be easy to miss in a code review. The fix is simple but the lesson is broader: every code path should return a value explicitly, input validation should happen at the boundary before any logic runs, and inconsistent patterns across similar operations are a maintenance risk even when they work correctly today.
Adding a log_operation decorator to TaskManager made it clear that decorators are just functions that take functions, once you see them that way the @ syntax stops being magic. The most important detail was functools.wraps, which preserves the original function's name and docstring so debugging and documentation don't break. The practical value is that cross-cutting concerns like logging, timing, and auth can be layered onto any function without touching its implementation.
The key shift from Python to JavaScript is that JS has no single obvious way to do things, where Python gives you one clean path, JS offers many competing patterns. Coming from Python, the biggest mental model adjustment is async: in Python it's opt-in via asyncio, but in JS the event loop is always running and everything is async-aware by default. Type coercion was the sharpest surprise == in JS silently converts types in ways Python never would, so === is the safe default, and ?? replaces Python's dict.get(key, default) more accurately than || which breaks on falsy values like 0. The most useful discovery was that JS array methods like map, filter, and reduce are more idiomatic than loops, mirroring Python's list comprehensions but as chained methods on the array itself.

The Repository pattern separates database queries from route logic behind a consistent interface, so routes never touch the database directly and swapping ORMs requires no route changes. Generic[T] makes the base Repository reusable for any model, Repository[User] gives a typed, model-aware repository without duplicating code per model. Dependency injection resolves bottom-up: get_db opens a session, get_current_user depends on both the token extractor and the session, and routes depend on get_current_user, each layer only knows about the layer below it.
When a request hits /admin/users/, the TimingMiddleware records the start time, FastAPI resolves dependencies bottom-up opening a DB session and decoding the JWT to fetch the current user, the requires_role wrapper checks is_superuser and raises 403 if false, the route queries the repository and returns the user list, and the middleware stamps the elapsed time onto the response header before it leaves.
The asynccontextmanager lifespan is a try/finally for the entire app, everything before yield runs on startup, everything after on shutdown, replacing the old @app.on_event pattern. JWT auth is stateless: the server signs a token containing the username and expiry, the client sends it on every request, and the server verifies the signature without storing any session. The logging system extends naturally from these patterns, record_action is a plain function called from routes after key events, and the audit endpoint reuses Depends(get_current_user) for auth the same way every other protected route does.
